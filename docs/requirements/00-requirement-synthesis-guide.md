# 需求综合指南

## 用途

使用本指南把原始输入转换成可直接实现的需求。

## 输入

- `docs/input/` 原始材料
- `docs/discussions/` 澄清记录
- 现有的 `docs/design/` 和 `docs/architecture/` owner 文档

## 输出预期

一份需求文件应清楚说明：

- 目标
- 范围
- 非目标
- 核心用户流程
- 业务规则
- 权限或角色影响
- 已知边界情况
- 未决问题
- 验收标准

## 规则

不要用漂亮但空泛的语言掩盖未解决的歧义，要明确写出来。

## 可实现门槛

只有在以下条件都成立时，需求才算可直接实现：

- [ ] 范围内行为具体到足以直接构建，不必猜测用户可见行为
- [ ] 非目标明确
- [ ] 描述了主要用户流程
- [ ] 角色/权限已覆盖，或者明确说明不相关
- [ ] 数据/模型影响已覆盖，或者明确说明不相关
- [ ] API/集成影响已覆盖，或者明确说明不相关
- [ ] 如果会对用户可见，空状态、加载状态、错误状态和权限拒绝状态都已覆盖
- [ ] 验收标准可测试
- [ ] 开放问题要么是非阻塞的，要么已经明确会阻塞实现

如果某个开放问题会改变用户可见行为、数据/模型形状、API 行为、认证/权限行为，或外部集成行为，请先暂停并请求澄清。

## 最小需求骨架

```md
# Feature: <name>

## Goal

## In Scope

## Out Of Scope

## Main User Flows

## Business Rules

## Roles / Permissions

## Edge Cases

## Open Questions

## Acceptance Criteria
```
