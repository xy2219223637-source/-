# 输入处理指南

## 用途

本指南说明原始源材料在变成需求之前应该如何处理。

## 规则

当原始输入 dump 仍然混合着以下内容时，不要直接让 AI 从中编码：

- 业务目标
- UI 示例
- 实现猜测
- 缺失的假设
- 只定了一半的范围

## 推荐流程

1. 把原始材料存入 `docs/input/`。
2. 标记来源类型：PM 笔记、卡片集文档、原型、文章，或混合来源。
3. 把未解决的问题写进 `docs/discussions/`。
4. 把综合结果写进 `docs/requirements/`。

## 有用的来源分类

- `source-pm-*.md` - 产品经理笔记
- `source-prototype-*.md` - 原型解读
- `source-cardset-*.md` - 卡片集或结构化需求文档
- `source-article-*.md` - 外部文章或参考资料

## 注意

结构清晰的源材料虽然有帮助，但它仍然可能无法回答：

- 当前迭代的真实范围边界
- 业务决策所需的领域判断
- 哪些交互是核心，哪些只是可选
- 原型是否已经足够完整，可以直接据此构建

## 文件头约定

放进这个目录的每个输入文件 SHOULD 以如下头部开始：

    status: new | supplement | supersedes <filename>
    processed: pending | partial | done

- `status` 描述该文件与其他输入的关系：
  - `new` - 一个独立的新输入，不替换已有文件
  - `supplement` - 对已有输入文件的补充上下文
  - `supersedes <filename>` - 替换先前的输入文件；被命名的文件现在已过时
- `processed` 跟踪该输入是否已被消化：
  - `pending` - 还未处理；代理应该优先看这个
  - `partial` - 部分问题已解决，但仍有未决项
  - `done` - 已完全转化为需求或讨论；代理可以跳过这个文件

如果人工放进来的文件没有这个头部，AI SHOULD：

1. 读取文件内容。
2. 结合内容和上下文推断状态。
3. 添加推断出的头部，并注明这是 AI 推断的。
4. 继续处理。
