# 审计执行指南

## 用途

本指南说明如何对需求、计划和实现进行独立审计。

## 审计原则

- 审计不是走流程，而是有意挑战当前说法
- 审计应引用文件、证据和 live repo 事实
- 审计应先列出发现，再给结论
- 不能只说“看起来不错”

## 计划审计

在实现前对计划进行审计，检查：

- 基线是否诚实
- 目标和非目标是否清楚
- 收尾门槛是否真实
- 是否存在隐藏依赖或未解决的需求缺口
- 是否有任何范围内缺陷或契约缺口被悄悄降级
- 任务路由和记录的技能使用是否诚实、必要且与 owner 文档匹配
- 待办池或上下文是否在没有人工确认或人工批准的 owner 文档证据的情况下被 AI 放宽
- 在实现前是否已对过时文档或遗留模式冲突进行分类
- 证明和验证是否覆盖每一条验收标准

## 收尾审计

在计划所追踪的切片完成后，独立检查：

- live 行为是否匹配当前需求
- 计划的收尾门槛是否全部满足
- 证明是否存在于文件和验证结果中，而不只是聊天里
- 支持基线变化处的文档是否已经更新
- 是否还有仍在范围内的缺口
- 任务路由和记录的技能使用是否仍然匹配已交付的工作
- 是否有任何自治或待办池状态是在没有持久证据的情况下被放宽
- 是否有验证失败或未运行命令被隐藏

## 结论格式

审计结果应先列出发现，并按严重度排序。

如果存在阻塞问题，写 `needs revision`，并列出精确要改的文件或章节。
如果没有阻塞问题，写 `passes plan audit` 或 `passes closure audit`，并列出剩余风险（如有）。
