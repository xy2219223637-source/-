# 先吸引子，后支架：面向大规模 AI 辅助开发的方法论

> 在 AI 深度参与开发的系统里，第一性问题不是“如何约束 AI 行为”，而是“系统长期应该收敛到什么结构”。
> 只有先把这个方向定义清楚，支架、护栏、验证、审计和收尾机制才真正有意义。否则，它们只是在更高效率地固化一个错误基线。
> 之所以必须强调这一点，是因为 AI 协作把软件工程里长期被当作外部附加项的“轨迹收敛”问题，推到了必须显式建模的位置。

## 1. 为什么谈 AI 工程，大家总是先谈支架

当前关于 AI 辅助开发的讨论，最常见的词是：

- `guardrail`
- `verification`
- `review`
- `feedback loop`
- `agent harness`

这类直觉默认了一个前提：

**系统的正确方向已经知道了。**

在这个前提下，问题自然会变成：

- 如何限制偏离
- 如何尽早暴露失败
- 如何让评审更严格
- 如何避免代理胡乱行动

这套语言足以应付小任务。对于脚本、CRUD 页面、局部 bug 修复，你通常已经知道正确结果应该长什么样，剩下的主要是执行和验证。

但在大系统里，真正的困难往往不是“如何不越线”，而是“哪条路才是真正的长期正确结构”。

要回答这个问题，先要看清一件事：**所有当前这类支架机制，其实都建立在一个隐含前提上，即评估单位主要是一个状态，而轨迹收敛则默认由人类的方向感来承担**。AI 协作恰恰移除了这个兜底机制。

## 2. 被移除的兜底机制

打开任何主流工程方法论，TDD、DDD、Clean Architecture、Agile，或者代码评审文化。它们的一等评估对象几乎都是状态：

- 评估“这个 PR 对不对”，而不是“过去 100 个 PR 累积出的方向对不对”
- 评估“当前架构清不清晰”，而不是“架构在演化过程中是否持续被推向稳定形态”
- 评估“测试是否覆盖当前行为”，而不是“测试套件本身是否已经被实现细节侵蚀”

传统方法论并不是完全不知道轨迹。重构、技术债、坏味道、演化架构、Lehman’s laws of software evolution 都涉及轨迹。但这些概念在体系中的地位是**修正机制和诊断词汇**，而不是基础对象。一个理论如果要不断借助“债务”“脆弱”“侵蚀”之类负向概念，事后才能识别轨迹问题，那恰恰说明它的基础范畴里缺少一个正向的轨迹对象。

这个以状态为中心的范式，过去之所以足够，是因为人类工程师本身就是低频、方向稳定的扰动源。一个人一天写几百行代码，每次扰动都很小，而且这个人会持续在脑中携带方向感。状态级质量保障，加上程序员的隐式方向感，就足以避免轨迹漂移。

AI 协作移除了这个兜底机制。AI 不是更快的程序员，而是**结构上完全不同的扰动源**：

- **高频**：一次会话可能在几分钟内生成跨多个模块的几百行代码
- **高幅度**：每次生成都可能跨边界引入结构性变化
- **没有持续方向感**：每个会话彼此独立，不具备跨会话的隐式架构判断
- **局部高度合理**：接口、类型、测试和文档可以一起生成，单独看每一项都可能通过检查

最后这一点尤其重要。**在 AI 协作里，所有状态级检查都可能通过，而整个系统仍然持续漂移。**

`nop-chaos-flux` 中的 Plan 76 就是典型例子。尝试移除 `array-editor` / `key-value` 的本地状态镜像后，直接触发了 11 个测试失败。但它暴露的并不是单个 bug，而是更深的一层事实：**测试本身已经被旧实现的时序耦合住了**。从状态视角看，每次累积变更都已经过了评审和 CI；从轨迹视角看，测试套件不知不觉滑到了无法支撑结构演化的位置。

传统理论有 `test fragility` 这个症状名，但那只是横切面的诊断，它没有命名“100 个合法提交如何累积成脆弱”这一过程。轨迹问题是过程问题，而过程问题需要过程语言。

所以问题并不是 AI 让旧问题更严重，而是 AI 把轨迹问题从**偶发修补对象**推成了**高频一等对象**。要把这类问题纳入方法论视野，工程方法必须把“轨迹”变成基础范畴。

## 3. 动力系统语言：让轨迹问题显式化

在数学物理里，**动力系统**是状态随时间连续演化、且下一状态依赖当前状态的系统。我们关心的不是某一时刻是否正确，而是它在时间上走出了什么**轨迹**。

在大规模 AI 辅助开发里，这套语言可以映射成四个彼此定义的基础对象：

- **状态空间**：在当前约束下，系统可能演化到的所有实现状态
- **吸引子**：系统在长期演化中会持续被拉回的稳定结构
- **轨迹**：每轮生成、验证和修正之后留下的真实演化路径
- **控制**：通过局部信号持续影响轨迹的机制

在当前约束下，仓库可能演化出的所有代码、文档和测试组合构成状态空间。人、AI、评审、CI、文档更新和其他持续动作构成演化规则。两者叠加得到的 live repository history 就是轨迹。吸引子则是系统在长期迭代中不断回归的稳定结构。

**这里必须说清楚：这套语言不是“更优雅地表达同一件事”，而是新的本体论语言。** 状态空间、吸引子、轨迹和控制四个概念彼此定义；少一个，其余三个就无法严格表达。若试图把这些概念翻译回旧本体论里的 architecture、constraints、goals、process，信息一定会丢失。这正是“AGE 不就是 harness 吗”这类误解的来源。

四个对象之间的关系是：

**state space -> attractor -> trajectory -> control**

**这不是修辞顺序，而是逻辑依赖。** 如果没有定义状态空间，就谈不上吸引子；如果没有定义吸引子，就没法判断轨迹是否漂移；如果没有轨迹判断，控制就没有目标。

这里的 “before” 指的是**逻辑优先**。在执行上，吸引子和支架会在循环里共同演化；但在概念上，可以先定义吸引子，再定义支架，而不能反过来。纠正应该朝向哪里？这正是吸引子与控制之间的非对称关系。承认共演化，并不会削弱“先于”的主张。终因和动力因在物理系统里也常常一起作用，但终因仍然是动力因的逻辑前提。

## 4. 吸引子到底是什么？

以著名的 Lorenz attractor 为例。它由微分方程隐式定义，不是一个提前列出所有正确轨迹的清单，也不是一个简单边界。吸引子内部的局部轨迹非常复杂，短期看甚至几乎像混沌；但整体并不会随机乱飞，而是持续被拉回到同一种几何形状。

![Lorenz attractor](https://gitee.com/canonical-entropy/nop-chaos-flux/raw/master/docs/ppts/assets/lorenz-attractor.png)

_Lorenz attractor：局部轨迹极其复杂，但整体仍受稳定结构约束。混沌不是随机；局部不可预测不等于全局失控。_

工程吸引子也是一样。**它更像“定义某个流形的方程”，而不是“列出所有合法点的清单”**。方程不会提前写出流形上的每一个点，它只规定哪些关系必须成立。满足这些关系的点，自然会收敛到同一结构。

**与传统架构概念的关键差异**：DDD 和 Clean Architecture 也强调长期结构，但它们通常把长期结构当作一个**目标状态**。方法论要做的是“达到 X”。轨迹本体论则把长期结构当作一个**吸引子**。方法论要做的是“无论被推得多远，都回到 X 附近”。**第二种视角处理的是扰动条件下的稳定性。** 这正是 AI 协作的核心问题，也是传统方法论缺少一等概念工具去处理的问题。

为了避免概念漂移，工程中的吸引子可以严格分成三层：

- **结构层，吸引子本身**：少量高层不变量，例如职责如何划分、边界如何建立、哪些结构关系不能被破坏
- **载体层，吸引子的工程载体**：把这些不变量外化成可版本化、可审计的文档
- **实现层，吸引子的瞬时投影**：当前代码里真正体现这些不变量的部分

**吸引子不是文档，文档只是吸引子的载体。吸引子不是代码，代码只是吸引子的瞬时投影。**

这个分层很重要，因为它解决了一个常见困惑：“当架构文档和代码冲突时，应该听谁的？”答案不是“谁更权威”，而是“你问的是什么问题”。如果你问当前实现行为，代码权威；如果你问系统应该往哪里收敛，文档权威；如果你问为什么某条路径被放弃，日志、bug 或分析权威。每一层只对自己对应的问题有权威性。

## 5. 仓库开始承载系统事实，支架就成了基础设施

AI 深度参与后，仓库不再只是人类认知的外部投影，它开始变成系统事实的主要载体。没人再完整掌握整个系统的设计细节。没人能不看项目源码就回答系统设计问题。下一次会话能重新加载的，不是作者脑中的完整意图，而是代码、diff、日志、测试和文档。

这带来一个直接的工程后果：

**生成和接受必须真正分离。**

生成动作可以由 AI 在同一上下文里快速完成；但接受不能再依赖这个生成上下文本身。必须回到仓库里的外部证据，再判断一次：

- 行为是否真的落地
- 当前基线到底是什么
- 哪些材料才是权威的
- 这次“完成”是不是只是完成感

正因为仓库开始承载系统事实，所以生成和评估必须分离。也正因为生成和评估分离，支架从“更谨慎的工程习惯”升级成了必要基础设施。

**在传统协作里，支架是可选纪律；在 AI 协作里，支架是让轨迹判断成为可能的必要条件。** 原因在于：AI 生成代码的同时，也会生成用来判断代码是否正确的材料，类型、测试、文档和完成总结都来自同一上下文、同一理解。如果这个理解本身有偏差，那么所有“验证证据”都会同向偏差。它们不会彼此矛盾，但整体可能是错的。这种**自我验证陷阱**在人工协作里会自然弱一些，因为 CI 规则、评审者和规范文档是独立于当前生成动作的外部标准，由不同认知主体维护。AI 协作打破了这种独立性，所以工程方法、fresh session、独立审计和来自 live repository 的证据收集，必须人为重建生成与接受之间的分离。

在仓库事实条件下：

- `test / lint / audit` 更像测量
- `owner doc / plan / closure` 更像约束
- `logs / bugs / discussions` 更像轨迹记录和外部化记忆

## 6. 最容易发生的三种本体混淆

第一，**把吸引子当成边界**。

边界回答“什么不能做”，一旦违反就会立即报错。吸引子回答“系统长期应该收敛到哪里”。单次违反不一定立刻报错，但持续偏离会腐蚀结构。边界定义禁区，吸引子定义稳定区。把两者混淆，会把吸引子降级成更严格的护栏，并错过它处理“扰动下稳定性”的核心能力。

第二，**把吸引子当成更强的护栏**。

护栏工作在执行层，每个动作都要检查。吸引子工作在方向层，判断累积动作是否在靠近它。把吸引子理解成更严格的治理、更密的约束或更强的审计，依旧只是把主要问题降级到控制层。根本原因在于没有意识到二者处于不同层级。

第三，**把吸引子当成控制目标的另一种说法**。

`control target` 这个词看起来接近，但它默认控制框架已经存在。在轨迹本体论里，吸引子的角色是**为控制提供终因**。它先于控制，而不是控制内部的一部分。若不先定义吸引子，所谓控制就没有目标，支架、护栏、验证和审计也就没有统一意义。

## 7. 在 `nop-chaos-flux` 仓库里，吸引子首先是 `docs/architecture/`

如果吸引子仍然抽象，它还没有真正的工程意义。对于 `nop-chaos-flux` 来说，最先承载吸引子的，是 **`docs/architecture/` 下带优先级的 owner 文档系统**。

在这个仓库里，工程落点非常明确：先有 `docs/architecture/` 下的规范定义，后面才有计划、验证、审计和日志等收敛机制。

在 `docs/architecture/` 内部，这一“方程层”也有明确优先级：

- `docs/architecture/README.md` 解释架构层级和阅读顺序
- `flux-design-principles.md` 负责方向层，解释设计意图和稳定原则
- `frontend-programming-model.md` 负责顶层规范层，定义原语身份、宏边界和硬不变量
- `flux-core.md` 负责当前全仓库基线
- 更窄的架构文档负责各自主题里的局部契约

在 `nop-chaos-flux` 中，吸引子的结构层不是抽象的“正确架构”，而是由少量高价值不变量共同定义的：七个原语的封闭集合、compile-first 管线、Template / Instance 分离、Data / Capability 正交、统一 renderer / hook 契约，以及 `flux-core -> flux-formula -> flux-compiler -> flux-action-core -> flux-runtime -> flux-react` 的依赖方向。

这些不是并列治理材料，而是由 owner 文档固定下来的结构方程。架构的价值不在于描述一切，而在于让错误结构无法继续合法存在。

正因为如此，这个仓库里很多真正重要的收敛动作，最后都会表现为“某种旧结构被排除出了合法状态空间”。

`CompiledSchemaNode` 的最终移除就是一个典型例子。它不仅仅是重构清理，而是在新的 Template / Instance 分离基线建立之后，旧中间结构虽然还能工作，却已经不属于正确结构，所以被排除。

同样，`flux-compiler` / `flux-action-core` / `flux-runtime` 的拆分也不只是“再拆出两个包”，而是说明吸引子已经更精确：系统不再只是“能跑”，而是进一步收敛成更稳定的职责结构。

## 8. 什么是支架？

如果吸引子解决的是“方向是什么”，那么支架解决的是：

**如何通过局部信号持续测量、修正和更新系统轨迹。**

这里的支架不是狭义的测试工具，而是更广义的执行骨架。它通常包括上下文路由、实现与接受的分离、计划和收尾条件、验证机制、审计机制、诊断工具和外部化记忆。

在这个仓库里，至少可以看到五层支架：

### 1. 路由支架

`docs/index.md` 和 `docs/architecture/README.md` 决定遇到问题时先读什么、当前基线是什么，以及什么只是分析、计划或历史。

### 2. 计划支架

`docs/plans/` 解决的是“这一轮扩张如何收尾”，而不是“系统是什么”。

`current baseline`、`goals / non-goals`、`exit criteria`、`validation checklist` 和 `closure audit evidence` 这些字段，定义了局部轨迹如何合法收尾。

在收尾/审计之后，Plan 145 又把新确认的后续面拆成 Plan 146。Plan 143 的收尾假设被新的审计反复推翻，直到 live repository 真正过线。这里的计划不是待办清单，而是局部收敛机制。它不只是列“现在要做什么”，而是规定这一轮扩张必须在哪里收尾、必须满足哪些退出条件，以及收尾需要哪些独立证据。

### 3. 验证支架

`lint`、`check`、`typecheck`、`build` 和 `test` 把高频、显式、可自动化的偏差检测下放到机器层。

### 4. 审计支架

并非所有偏差都能被自动规则捕获。更高层的语义漂移、结构偏差、假完成，以及局部自洽但整体失真，仍然需要独立审计。

审计本身也是闭环：发现偏差，过滤冲突和假问题，然后回到 live repository 重新确认。高质量审计不是堆更多发现，而是尽快排除无效发现。

在这个仓库里最重要的支架规则之一是：**不要让同一个上下文既实现又裁定完成**。完成必须来自 fresh session 或独立审计，它们要重新检查 live repository。Plan 143 和 Plan 145 的收尾之所以有分量，正是因为“完成”不是实现者自报，而是独立收敛判断。

### 5. 记忆支架

`docs/logs/`、`docs/bugs/` 和 `docs/discussions/` 构成跨会话的外部化记忆。

这一层尤其重要，因为对 AI 而言，“为什么不能那样理解”本身就是系统记忆的一部分。只保留最终结论是不够的。很多时候，还必须保留：

- 哪个前提已经被证伪
- 哪条路径已经被证明会偏离
- 哪种术语翻译会把问题降级
- 哪个“已完成”判断后来被 live repository 推翻

没有这个记忆支架，每次会话系统都会丢失一部分历史轨迹信息。

真实循环不是“定义一次吸引子，然后永远执行支架”，而是“定义吸引子 -> 扩张 -> 修正 -> 更新吸引子 -> 再扩张”。`flux-compiler` / `flux-action-core` / `flux-runtime` 的三层拆分，就是吸引子被实践修正之后继续扩张的例子。

## 9. 为什么新吸引子通常不会由 AI 自己演化出来

即使有支架，也不能指望 AI 在高速迭代中自己慢慢演化出一个新的吸引子。

至少在当前主流模型的训练分布和偏差下，一般不应该期待它自然发生。

更具体地说：

- 当前主流 AI 很擅长围绕既有吸引子高速扩张和收敛
- 但它通常会回到自己见过的平均解
- 真正新的概念切分、边界重定义和架构语言，仍然需要人类先提出来

**定义新吸引子的责任，默认不能外包给 AI。**

在大型框架开发中，人和 AI 的分工不能简单理解成“AI 写代码，人类评审”。更现实的分工是：

- 人类定义新的吸引子
- AI 在既有吸引子周围快速扩张
- 支架持续把轨迹拉回正路

那些真正改变系统结构语言的概念，比如 `ActionScope` 和 `Data Scope` 的分离，以及词法作用域，不可能只靠让 AI 自由采样平均解长出来。它们必须先作为新吸引子被人类提出，然后 AI 才能在新基线上做大规模扩张。

这里需要承认一个限制：**这意味着方法论的有效性依赖于团队里有人能定义吸引子**。这不是方法论的弱点，而是它正确识别到的事实：架构判断是一种稀缺资源。传统协作里，这种判断可以部分留在架构师脑中，通过口头说明和代码评审传递；AI 没有跨会话记忆，任何没有显式外化的架构判断，对 AI 来说就等于不存在。这个方法论的工程贡献，正是把这种稀缺资源外化成可版本化、可审计、可继承的仓库结构，让它不随着某个人离开而消失。但如果团队里根本没人能做这种判断，它也不可能凭空生成。

## 10. 结论

大规模 AI 辅助开发真正困难的部分，不是让 AI 写更多代码，而是让系统在高速扩张时持续收敛到正确结构。

在以状态为一等公民的传统方法论里，这种收敛问题没有正向对象，只能通过“债务”“脆弱”“侵蚀”之类负向词汇回头识别。要把收敛变成可表达、可讨论、可工程化的一等对象，方法论就必须把轨迹纳入基础范畴。

谁提出新的吸引子，谁就定义了系统后续演化的结构基线。

压缩成一句话，这个方法的核心就是：

**state space -> attractor -> trajectory -> control.**

这不是更复杂的工程流程，而是更基础的工程概念结构。只有当系统长期应该收敛到某个结构时，我们才能判断轨迹是否漂移；只有能判断轨迹漂移，支架、护栏、验证、审计和收尾才有统一意义。

在 AI 时代，真正稀缺的不是更多能写代码的代理，而是能先回答“系统应该收敛到哪里”的人。
