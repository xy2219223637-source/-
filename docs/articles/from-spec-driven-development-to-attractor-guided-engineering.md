# 从规格驱动开发到吸引子引导工程

`nop-chaos-flux` 是一个低代码运行时框架，使用 React 19、Zustand 5 和 Vite 8 构建。它受 Baidu AMIS 启发，但完全是围绕创新设计原则从零独立重写的。它包含 JSON Schema 编译器和运行时，以及 Report Designer 和 Flow Designer 等设计工具。

这个项目最初由一名程序员在两个月内，完全借助自动化 AI 辅助开发完成。与普通的 AI 辅助开发不同，这个项目并没有随着功能增加而出现质量退化。相反，模块边界变得更清晰，文档分类更稳定，测试和审计更擅长捕捉真实问题，代码质量也持续提升。

我最近在几场内部分享中把这种实践命名为 **Attractor-Guided Engineering**（AGE）。后来在代码评审中我发现，很多人其实并没有真正理解它。有些人的工作流更接近随意的 vibe coding。另一些人则更像是在用 OpenSpec 风格的规格驱动开发，整体上倾向于按任务、按功能逐个推进。随着工作不断累积，代码中的不一致也不断累积，偏离架构设计逐渐变成常态。

本文分析 AGE 与规格驱动开发之间的差异，以及普通业务项目能从 `nop-chaos-flux` 的大规模 AI 工程实践中学到什么。

## 核心层级

AGE 的核心概念层级是：

**state space -> attractor -> trajectory -> control**

- **状态空间**：在当前约束下，系统可能演化出的所有实现状态
- **吸引子**：系统在长期演化中会反复被拉回的稳定结构
- **轨迹**：每轮生成、验证和修正之后留下的真实演化路径
- **控制**：通过局部信号持续影响轨迹的机制

在现有约束下，仓库可能演化出的全部代码、文档和测试组合构成状态空间。人、AI、评审、CI、文档更新以及其他持续动作共同构成演化规则。两者叠加后的 live repository history 就是轨迹。吸引子则是系统在长期迭代中不断回归的稳定结构。

大规模 AI 开发，本质上是动力系统中的受控收敛问题。AI 会极快地扩展状态空间。关键不是到处增加护栏，而是先理解系统长期应该被拉回到什么结构。

吸引子不是一个写好的终点，也不是路线图，更不是允许活动的边界。它是一种由少量高层约束隐含定义出来的结构：局部实现可以变化，但整体会持续被拉回到同一类形状。

在 `nop-chaos-flux` 中，最先定义吸引子的不是计划、lint 规则、测试、审计或 spec 文件，而是 `docs/architecture/` 下带有明确优先级的架构文档。本文把这些带有 owner 和优先级的架构文档称为 owner docs。

`docs/architecture/README.md` 明确说明，`docs/architecture/` 是当前最终状态的架构基线，`flux-design-principles.md` 是方向原则锚点，`frontend-programming-model.md` 负责顶层原语和核心边界优先级，其他规范性文档则在各自主题内拥有局部优先级。

这些文档并不会枚举所有正确实现。它们更像方程：定义了有哪些原语、哪些依赖方向合法、哪些 owner 边界不能被打破，以及哪些旧模式已经不属于正确状态空间。它们先定义出一个可持续的状态簇，之后实现、测试、计划和审计才有参照系。

计划、验证、审计、日志、bug 和测试都在后面。它们不是吸引子本身，而是维持系统轨迹接近吸引子的支架。

Flux / AGE 不以 spec 演化为中心，而以吸引子定义和轨迹收敛为中心。

## 真实开发历史

“架构文档定义吸引子”乍看起来像普通的文档治理。但如果你仔细看 `docs/logs/` 下的真实开发记录，差别就会变得非常明显。

例如，`docs/plans/371-deep-audit-2026-05-19-owner-routing-plan.md` 就是一个典型入口。它并没有直接改代码，而是把 `2026-05-19` 那次深度审计保留的 64 条发现，收敛成了一个 owner-routing 基线。每条发现都有唯一的 owner bucket、优先级、后继计划和 owner 文档义务，没有无人认领的问题，也没有多重归属问题，更没有把问题悄悄降级成含糊的后续项。

它的作用，是冻结一轮偏差应该如何收尾。后续执行计划再把局部轨迹收束到具体结果面。`docs/plans/382-deep-audit-2026-05-19-table-and-crud-owner-state-and-event-contract-plan.md` 就关闭了 table / CRUD 的 owner state 和 event payload：`Current Baseline` 说明了显式空数组 owner state 和 event payload 的 live 状态，执行项修复了空数组回退，聚焦测试证明了结果，`docs/components/table/design.md` 和 `docs/components/crud/design.md` 同步了 owner 契约，最后仓库级别的 `pnpm typecheck`、`pnpm build`、`pnpm lint` 和 `pnpm test` 全部通过。独立收尾审计 `ses_1bd9ed593ffeVpkho4lb4wPR6p` 记录了 `Verdict: acceptable`。

`docs/plans/388-deep-audit-2026-05-19-form-tree-widget-accessibility-plan.md` 说明了另一种收尾方式。input-tree 和 tree-select 的可见节点 roving focus、`ArrowUp` / `ArrowDown` / `Home` / `End` 导航，以及加载态的 `aria-busy` / `aria-describedby` 都有聚焦证明，但计划明确裁定 `No owner-doc update required`。这说明计划不只是任务清单，它也要负责判断这次变更是否会修改 owner 文档基线。

`docs/plans/400-deep-audit-2026-05-19-test-harness-reliability-plan.md` 和 `docs/bugs/62-e2e-shared-websocket-error-suppression-fix.md` 体现了记忆支架的作用。问题并不是某个业务功能失败，而是 E2E 共享 fixture 的过滤过宽，把真实的 transport/runtime 失败也压掉了。修复不只是改代码，还增加了 `tests/e2e/fixtures-hard-gate.spec.ts` 来证明 fixture 现在会失败，并记录了一条 bug 规则：如果未来测试确实需要容忍 WebSocket 失败，必须使用逐测试的放行，而不能恢复成 fixture 级的压制。

`docs/logs/2026/05-19.md` 和 `docs/logs/2026/05-20.md` 并不是日记，而是记录聚焦证明、owner 文档裁定、仓库级门禁、独立审计和全绿基线的闭环证据。

这些材料连成一条真实链路：owner 文档定义长期结构；审计发现偏差；计划负责路由并冻结归属；聚焦证明展示局部结果；owner 文档裁定决定是否需要更新基线；日志、bug 和测试保存跨会话记忆；收尾审计重新判断完成是否成立。

这就是 AGE 中的 harness。目标不是让每个任务看起来更完整，而是让系统轨迹不断回到吸引子。

## OpenSpec 为什么结构上很强

OpenSpec 是一个代表性的规格驱动开发框架。它对一类工作组织得很好：

- `openspec/specs/` 保存能力的当前行为规格
- `openspec/changes/` 保存提议的变更，其中包括 `proposal.md`（为什么改、改什么、影响什么）、必要时的 `design.md`（技术设计）和 `tasks.md`（实现清单）
- `changes/<name>/specs/` 记录规格增量，使用 `ADDED`、`MODIFIED`、`REMOVED` 和 `RENAMED`
- `archive` 按固定规则把增量写回主规格

这套机制降低了维护规格的成本。OpenSpec 不要求 AI 每次都自由理解完整需求再任意改文档；它通过固定 section、需求标题匹配和 delta 应用，把结构化变更重新合并回主规格。这在行为规格清晰、并且需要可解析、可归档、可一致回写的场景里很有效。

## OpenSpec 为什么受限

OpenSpec 的规格格式有利于工具处理，但作为组织项目知识的通用方式，它并不够灵活。

很多仓库知识并不天然长成 requirement / scenario 的样子：

- 架构层级和优先级
- 术语被误读的历史
- 外部研究材料，例如 PPT 和讨论记录
- 一次源码审计的结论
- 手动测试的观察
- 复杂 bug 的诊断路径
- 为什么某个计划收不住尾

如果这些东西都必须先翻译成 spec/change 形式，才能得到正式位置，那么转换成本会很高，而且原始上下文会丢失。外部文档、讨论记录、分析报告和 bug 回顾，完全可以直接成为仓库记忆；强行压进需求/场景，会把信息压扁。

在执行验证方面，两种方法也体现了不同的工具哲学。OpenSpec 的 `tasks.md` 是清单，`/opsx:verify` 是一个可手动开启的验证技能，必须通过扩展工作流才能启用。`archive` 只检查任务状态并提示风险，但用户确认后仍可继续。严格收尾留给团队自己。

Flux 计划则把独立收尾审计放进了必经路径。计划必须从当前基线出发，写明 `Goals` 和 `Non-Goals`，把执行项标注为 `Fix` / `Decision` / `Proof`，并明确 `Closure Gates`。在被标记为 `completed` 之前，独立子代理或评审者必须回到 live repository 做收尾审计。`docs/plans/361-slot-contract-host-manifest-and-owner-doc-closure-plan.md` 把当前基线、发现归属矩阵、执行阶段、收尾门槛、已裁定延后项和独立评审证据放在一起。这不是清单，而是严格定义完成条件，并要求独立审计证据。它强调的不是“做了什么”，而是“必须落地到什么程度，才真的算数”。

## 为什么规格驱动开发容易变成任务驱动

规格驱动开发很自然会把人和 AI 的注意力拉回到“如何完成这次变更”。

一次变更通常会有提案、设计、任务和 delta specs。它比口头需求强得多，但天然仍然围绕单次变更组织：为什么改、改什么、怎么做、哪些清单项完成了。随着时间推移，它很容易变成一种更正式的任务分发系统。

这种任务化有一个隐蔽风险：规格已经更新、任务已经勾掉、archive 已经跑过，于是人和 AI 都很容易产生强烈的“已经完成”感。但系统是否真的更接近长期结构，并不会因为这次变更本身而被证明。这个问题必须回到架构文档、live repo、测试、日志和独立审计里再检验。

如果你想把规格维持到 Flux 级别的精度，成本通常会更高。Flux 不维护单一 spec tree，而是维护一组职责不同的仓库记忆：架构文档定义吸引子；计划定义局部轨迹收尾；日志记录演化；bug 保存复杂诊断；测试保存人工发现；分析保存研究判断；审计重新检查 live repo。把这些东西都塞进 spec/change 结构里，会带来持续的转换成本。

规格驱动开发可以作为 AGE 内部管理行为规格演化的局部支架，但它不应该替代架构文档，也不应该替代计划收尾、日志、bug、测试和审计这些独立仓库记忆。

## 如果你只需要读一份 owner 文档呢？

这恰好说明两种方法的根本差异。

在 Flux 中，如果你只需要查看 renderer runtime 的当前 owner 契约，可以直接从 `docs/index.md` 路由到 `docs/architecture/renderer-runtime.md` 及其相关引用。不需要创建变更、不需要写提案、不需要定义任务，也不需要把问题改写成 spec delta。

如果只是研究，就放到 `docs/analysis/`；如果是手动测试发现，就放到 `docs/testing/`；如果是复杂 bug，就放到 `docs/bugs/`；如果是执行收尾，就写计划；如果稳定基线变化了，就回到 owner 文档。

Flux 的文档组织是自由的，但不是任意的。自由，意味着它不会强迫所有知识先经过同一种工件流程；有约束，意味着每种材料都有清晰职责，不能随便自封事实权威。

OpenSpec 的组织更结构化，适合把行为规格变更组织成变更包，再把 delta 写回主规格。但它不是组织一般仓库知识的默认主线。

## 任务不是计划

OpenSpec 的 `tasks.md` 并不对应 Flux 的 `plans`。

最多只能说，`tasks.md` 相当于 Flux 计划中的执行列表碎片，或者当前会话的待办清单。它可以帮助 AI 不漏步骤，但它回答不了这些问题：

- 当前 live repository 的基线是什么？
- 这次变更明确不做什么？
- 什么证据证明工作真的完成了？
- 哪些缺陷不能被降级成后续项？
- 谁来独立复核完成？

Flux 计划的价值不在于“任务更多”，而在于它是局部轨迹的收尾契约。

因此，任务很有用，但任务不是事实来源。`- [x]` 只表示实现者声称某项已经完成。真正的证明必须回到当前代码、测试、owner 文档和独立评审。

## 变更来源不只有规格变更

OpenSpec 的路径很清晰：围绕 spec 和 changes 组织行为规格演化。

Flux 的变更来源更广：

- 用户提出的新需求
- 通过读源码发现的架构漂移
- 审计发现的 owner 边界问题
- 测试暴露的交互问题
- bug 笔记中记录的历史回归风险
- 外部研究或分析带来的设计判断
- 计划收尾审计推翻了此前“已完成”的判断

这些来源不一定要先变成 spec delta。它们可以先进入分析、测试、bug、日志、计划，或者直接更新架构文档。是否需要最终变成行为契约，取决于它是否属于长期基线。

这正是 Flux 文档组织自由度的意义：不同来源的信息先停留在不同位置，而不是全部塞进同一个规格演化通道。

## 为什么 file-in / file-out 很重要

我在很多分享里都强调，AI 驱动开发的一个最佳实践是 file-in / file-out。这不是为了文档洁癖，而是因为吸引子和支架都需要载体。

File-in：输入不要只停留在聊天窗口。即使内容很乱，也先写进需求文件、分析文件或计划文件，然后再把这个文件 `@` 给 AI。

File-out：输出不要只打印在窗口里然后消失。重要结论、分析、计划、测试记录、bug 诊断和架构约束，都应该写进 `docs/` 并按职责分类：

- 架构规则 -> `docs/architecture/`
- 变更收尾 -> `docs/plans/`
- 执行轨迹 -> `docs/logs/`
- 手动测试发现 -> `docs/testing/`
- 复杂缺陷诊断 -> `docs/bugs/`
- 研究判断 -> `docs/analysis/`

重点不是“写更多文档”，而是把输出分类并持久化后，让它进入仓库记忆。聊天窗口只是临时上下文，文件才是仓库记忆。

## 代码是当前实现事实，但不是唯一事实

代码是当前实现的事实来源。类型和测试保护当前行为。带优先级的架构 owner 文档定义吸引子，也就是系统长期应该收敛到的结构。日志、bug、分析和测试则记录轨迹和外部化记忆。

这些并不冲突。冲突通常只在两种情况下出现：

- 文档开始重复脆弱的实现细节
- 把文档完成当成代码已经正确的证明

OpenSpec 的行为优先边界，也是为了避免第一种问题：行为规格应该描述可验证的行为，而不是内部实现细节。Flux 进一步强调第二种问题：即使文档陈述的行为是对的，它也不能替代当前仓库的验证。真正的收尾必须回到代码、测试、owner 文档、日志和审计证据。

## 真正的区别

OpenSpec 解决的是：如何通过结构化的 spec、change、delta 和 archive 来组织行为规格更新。它关注的是规格的演化流程。

AGE 解决的是：当 AI 以很高速度扩展状态空间时，系统如何先定义吸引子，再利用计划、验证、审计、日志、bug、分析、测试和其他支架，让轨迹持续收敛。

这两个工具处于不同层级。OpenSpec 擅长结构化规格工作流；Flux / AGE 擅长通过带优先级的架构 owner 文档定义吸引子，再用自由但有职责边界的文档路由和严格的收尾支架，让仓库持续收敛。

从规格驱动开发走向 Attractor-Guided Engineering，并不是从“写规格”退回到“不要写规格”，而是从“把变更组织成规格更新”转向“先定义系统长期应该收敛到哪里，再把每轮 AI 生成拉回这个结构里”。

## 普通项目可以直接使用什么

你不需要复制 Flux 的完整治理。

**第一，file-in / file-out。** 在把输入 `@` 给 AI 之前先把它放进文件。把输出写进 `docs/`，不要只留在聊天窗口里。这个动作只需要几分钟，却能让 AI 依赖可追踪的仓库记忆，而不是上下文记忆。

**第二，给核心业务写一份架构文档，并定义业务吸引子。** 不要写代码导览。要写事实来源、状态转换、归属边界和常见误解。

**第三，为复杂变更写轻量计划。** 计划里要包括当前基线、目标、不做什么、执行项、证明项和收尾条件。完成前，用独立会话或独立评审重新核查。

**第四，把重复问题变成仓库记忆。** 手动问题放进 testing，复杂 bug 放进 bugs，研究结论放进 analysis，高频错误放进审计脚本。

spec 更新仍然是必要层，但系统收敛还需要吸引子。
