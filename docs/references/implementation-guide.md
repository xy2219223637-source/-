# 实现指南

## 用途

本指南记录 AI 和人类在日常编码中的期望。

## 核心规则

- 优先做最小且正确的改动
- 先遵循现有模式，再发明新抽象
- 保持状态所有权显式
- 避免只用于演示的占位符，除非当前需求明确允许
- 当 bug 或契约可能悄悄回归时要加测试
- 除非计划文件里确实需要实现细节来解释范围、证明或契约，否则不要把代码级技术细节写进去
- 默认尽量不写注释；只有在局部约束很容易被误读时，才少量添加高价值注释

## 评审视角

在评审工作时，检查：

- 边界是否被越过
- 是否有未写进文件的隐含假设
- 是否仅凭 UI 形状或聊天说法就产生了假完成
- 用户可见行为是否缺少验证

## 协作规则

如果某条重复指令不断出现在聊天里，就把它提升到 `AGENTS.md`、`docs/architecture/` 或 `docs/references/`。

如果某个重复的评审模式总是发现同一类问题，就把它提升到 `docs/skills/` 或 `docs/audits/`。

如果同一种缺陷模式在那之后仍然不断出现，就评估是否进一步提升为启发式脚本、静态检查、lint 规则、CI 守卫或 codemod。优先使用轻量检查，并把它们调到适合被复制项目真实约定、命名模式和误报容忍度的程度。
