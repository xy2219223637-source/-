# 项目约定

## 用途

本文件记录 AI 代理默认应遵守的项目级规则。

保持简短。如果某条规则变成了详细的查找材料，就把细节移到 `docs/references/`，然后在这里链接过去。

## 文件进出

- 重要输入应先写入文件，再开始实现。
- 重要输出应写回仓库，而不是只留在聊天里。
- 原始输入属于 `docs/input/`。
- 综合后的、可直接实现的需求属于 `docs/requirements/`。

## 设计拆分

- 需求/应用行为设计属于 `docs/requirements/` 和 `docs/design/`。
- 技术架构设计属于 `docs/architecture/`。
- 如有重复规则，优先交叉引用，不要在多个文档里重复写同一条。

## 评审规则

- 高风险或高歧义的需求和设计草稿，应由独立子代理或评审者再过一遍。
- 任何创建的计划都必须先通过独立计划审计，再通过收尾审计。
- 独立评审应引用文件和证据，而不只是说“看起来不错”。
- 如果没有独立评审者可用，应在计划或日志里记录这个限制。冷重放不是第二位评审者，也不能单独解决受保护区域或事实来源冲突。

## Bug 规则

- 每个非平凡 bug 修复都应该增加或更新自动化测试覆盖。
- 如果无法自动化覆盖，请记录原因以及人工证明方式。

## TDD 原则

- 功能开发遵循测试先行：先写测试，再写实现代码。
- 单元测试（Unit Tests）：针对单一函数/组件，在代码实现前编写，确保最小单元的正确性。
- 集成测试（Integration Tests）：针对模块交互，在接口定义后编写，验证组件协作。
- API 测试（API Tests）：针对外部接口契约，在设计确认后编写，保障前后端/服务间协议稳定。
- 任何新功能提交前，必须确保对应测试通过。

## 注释策略

- 默认尽量不写注释。
- 只有当某个局部约束很容易被误读，而代码本身又不足以说明时，才添加注释。

## 验证规则

- 在 `docs/context/project-context.md` 中保持验证命令最新。
- 没有真正运行过的命令，不要报告验证成功。
- 复制模板后，不要把占位符验证命令一直留着。
