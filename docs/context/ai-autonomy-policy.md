# AI 自治策略

## 用途

本文件定义 AI 代理何时可以不询问直接继续，何时必须停下来等待人工输入。

保持简短且贴合项目。只要团队希望 AI 更主动或更保守，就更新这里。

AI 可以把这个文件改得更严格，也就是把工作限制得更细；但 AI 不能在没有明确人工确认或带有人类批准证据的 owner 文档前提下，放宽受保护区域，把 `ask-first` / `blocked` / `research-only` 的工作改成 `implement`，或移除阻塞项。

AI 编写或修改的文档，包括 owner 文档，不能作为放宽自治、清除阻塞、把文档标成新鲜，或降低受保护区域级别的证据，除非人工明确批准这些证据。

## 自治级别

在 `docs/backlog/` 和活动计划中使用这些标签：

- `implement` - 只要读完列出的需求、owner 文档和验证命令，AI 就可以实现。
- `plan-first` - AI 可以起草或更新计划，但实现要等计划审计以及下表要求的受保护区域批准。
- `ask-first` - 在修改代码或用户可见行为之前，AI 必须先询问。
- `research-only` - AI 可以检查、总结和提出选项，但不得修改产品行为。
- `blocked` - 在阻塞项通过文件或人工确认解决之前，AI 不得继续。

## 评审可用性

给复制后的项目设置一个值：

- 评审可用性：`<human | subagent | none>`

如果这个值仍然是占位符，请把评审可用性当作 `none`，并把受保护区域或高风险计划视为 blocked，直到配置好 human/subagent 评审。

规则：

- `human` 或 `subagent` - 使用该评审者执行所需的计划审计和收尾审计。
- `none` - 只有在非受保护、非高风险的计划中才可以使用冷重放。冷重放不是第二位评审者，它只是把实现上下文放下后再做一次有记录的自检。
- 受保护区域、未解决的产品风险或事实来源冲突，仍然需要 human/subagent 评审，否则必须保持阻塞。

## AI 可以在不询问的情况下继续，当且仅当

- 工作项标记为 `implement`，或者用户直接要求一个本地低风险变更
- 当前需求具有可测试的验收标准
- 当前 owner 文档已经列在 `docs/context/project-context.md`
- 对于从待办池选择的工作，待办行是 `ready`，没有过期链接，并且不需要缺失的计划
- `docs/context/project-context.md` 中的验证命令是真实命令，不是占位符
- 本文件中的受保护区域占位符已经替换为真实条目或明确的 `none`
- `docs/context/project-context.md` 中的文档新鲜度是 `fresh`，或者当前切片已经明确验证过需求、owner 文档、codebase-map 路由和受影响代码区域都是新鲜的
- 任务不触及下列受保护区域
- 开放问题已经明确标记为非阻塞

## AI 在以下情况前必须询问或停下

- 需求或 owner 文档不清楚，却要修改产品范围
- 在没有 owner 文档和测试策略的情况下，修改数据库/模型形状、数据删除、支付、认证、权限、部署或外部集成行为
- 根据未写入已提交集成文档或测试中的外部系统，臆造其行为
- 因为命令缺失、损坏或太慢而跳过必要验证
- 关闭一个仍缺少审计、验证、文档或检查单证据的计划
- 当 live code 与 owner 文档冲突，而且解决冲突会改变用户可见行为或公开契约时，继续推进
- 在没有人工确认或人工批准的 owner 文档证据时，放宽自治标签、受保护区域规则或阻塞项
- 当当前切片的文档新鲜度为 `stale`、`unknown` 或 `partially stale` 时继续实现；应先做基线研究或 plan-first 对齐切片

## 受保护区域

为复制后的项目补全这些内容。

如果这个表格仍然包含占位符，AI 必须把支付、认证/权限、数据删除、数据库/模型形状、部署以及外部集成视为 `ask-first` 或 `blocked`，直到表格被真实条目或明确的 `none` 替换。

| 领域 | 规则 | 所需证据 |
| ---- | ---- | -------- |
| `<payment>` | 先询问 | owner 文档 + 测试 |
| `<data deletion>` | 先询问 | owner 文档 + 测试 |
| `<auth/permissions>` | 先计划 | owner 文档 + 测试 |

受保护区域规则含义：

- `ask first` - 在规划或实现之前必须得到人工批准。
- `plan-first` - AI 可以起草计划，但实现需要计划审计以及表中要求的证据。如果评审可用性是 `none`，实现仍然阻塞。
- `research-only` 或 `blocked` - AI 不得修改产品行为。

## 待办池选择规则

如果用户要求 AI 继续工作但没有指定任务，就选择 `docs/backlog/README.md` 中优先级最高、自治级别为 `implement` 且阻塞项为 `none` 的工作项。

在实现所选工作项之前，重新检查规划触发条件。`Plan: none` 并不意味着可以绕过计划指南。

用户直接要求的本地低风险编辑不需要待办池行，但仍然必须满足免计划路径和验证规则。

如果不存在安全的 `implement` 项，请总结当前最阻塞、`plan-first` 或 `ask-first` 的项，并请求决策。
