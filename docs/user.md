# 用户认知能力画像

## 基础背景

- 身份背景：中国理科生。
- 高中知识结构：语文、数学、英语、物理、化学、生物。
- 英语能力：读写约大学英语四级水平。

## 能力等级说明

- 了解：知道基本概念、用途和大致工作方式。
- 入门：能跟随教程或示例跑通基础流程，能做简单修改。
- 可用：能独立完成小型任务或项目模块，能进行基础排错和组合应用。
- 熟练：能独立设计方案、处理复杂问题、优化工程实现，并能指导他人。

## 设计认知

- 设计需要区分为事实性设计和过程性计划。
- 事实性设计：描述系统、业务、数据、概念、约束、关系、状态等“是什么”的内容。
- 过程性计划：描述步骤、执行顺序、任务拆分、里程碑、验证方式等“怎么做”的内容。
- AI 在协助设计时，应避免把事实性设计和过程性计划混在一起。
- 对复杂任务，优先先澄清事实性设计，再制定过程性计划。

## 测试与质量偏好

- 了解 TDD。
- 倾向在开发前置阶段设计测试。
- 常规前置测试类型：
  - 单元测试。
  - 集成测试。
  - API 测试。
- E2E 测试：与单元测试、集成测试、API 测试分开处理。
- E2E 执行原则：E2E 成本高、速度慢，应在全部 API 测试结束并通过后再执行。
- E2E 工具倾向：Playwright CLI。
- 测试协助工具：
  - OpenCLI：可用于辅助测试流程。
  - PageAgent：可能对页面级自动化测试有帮助。

## 当前优势

- Python 与 FastAPI 可用，适合先以后端 API 和 AI 应用编排作为主线，再逐步补齐 React 前端能力。

## 当前短板

- 前端能力暂弱，React 需要系统补齐。
- 多个 AI 工具虽然达到可用层级，但需要通过项目沉淀形成稳定方法论。
- 英语读写可支撑技术资料阅读，但仍可继续提升英文文档阅读速度和技术表达能力。
- 本体论与知识图谱方向已有概念基础，但需要更多业务建模实践。

## 建议学习路线

1. 巩固 Python + FastAPI + 测试基础，形成稳定后端开发能力。
2. 使用 React 补齐基础前端能力，优先掌握组件、状态、表单、请求、路由。
3. 以 Dify/Coze 快速验证业务流程，以 LangChain/LangGraph 承接更可控的工程实现。
4. 以 LightRAG、GraphRAG、Neo4j、Milvus 构建知识增强应用。
5. 在每个项目中前置测试设计，优先完成单元测试、集成测试、API 测试；全部 API 测试通过后，再执行成本更高、速度更慢的 E2E 测试。

## AI 使用本文件的原则

- 本文件用于帮助 AI 了解用户的认知范围、能力边界、协作偏好和可能的认知局限。
- 用户认知范围内的内容，可以直接讨论、推演、实现和迭代。
- 用户认知范围之外的内容，AI 默认可以根据任务目标和工程常识自行决定，并在结果中简要说明关键判断。
- 当用户认知范围之外的内容存在高风险时，AI 应详细解释原因、影响、备选方案，并获得用户批准后再执行。
- 高风险情况包括但不限于：修改自身配置、修改运行环境、可能导致任务中断、可能破坏已有数据、可能影响账号权限、可能产生费用、可能引入安全风险。
- 对低风险、可回退、局部性的实现细节，AI 可以自行决定，不必频繁打断用户确认。
- 当 AI 判断用户可能误解了关键概念、成本、风险或执行顺序时，应主动解释。

## 与 AI 助手协作偏好

- 希望先给出清晰结构，再进入实现。
- 遇到新技术时，优先使用最小可运行示例。
- 写代码时倾向配套测试。
- 复杂项目希望拆成需求、设计、实现、测试、验收几个阶段。
- 对不熟悉的前端部分，希望 AI 助手给出可运行实现并解释关键结构。
